# Mvp Wireframes

**Catégorie** : Idéation  
**Durée** : 60-90 min

## Pourquoi

Les wireframes permettent de valider l'UX avant de coder. C'est 10x moins cher de corriger sur papier qu'en production.

## Objectif

Créer les wireframes de ton MVP.

## Livrable attendu

Wireframes des écrans principaux

## Étape 0 — Contexte (obligatoire)

Charger `./rapido-kb/` — surtout `./rapido-kb/startup/` (vision, persona,
offre, hypothèses construits par `dossier-startup-360` et
`interview-business-plan`). La KB PRIME : ne jamais redemander une
information déjà validée, la reformuler pour confirmation. Les chiffres
réels viennent des MCP — jamais de mémoire ; tout chiffre sans source datée
porte la mention « hypothèse fondateur, confiance faible ».

## Étapes

1. **Lister les écrans** — Home, signup, dashboard, etc.
2. **Prioriser le user flow** — Le chemin critique d'abord
3. **Sketcher low-fidelity** — Papier ou outil simple
4. **Créer mid-fidelity** — Avec Relume, Figma ou Whimsical
5. **Tester avec des users** — 5 tests suffisent pour 80% des problèmes

## Pro tips

- Commence par mobile-first
- Un écran = un objectif
- Teste avec de vrais utilisateurs le plus tôt possible

## Erreurs fréquentes

- Trop de détails trop tôt
- Pas de hiérarchie visuelle
- Oublier les états (loading, error, empty)

## Données & serveurs MCP

- **RapidoCRM** (`rapidocrm`) — données réelles, lecture d'abord
- **Lovable** (`lovable`, via le plugin `rapido-lovable`) pour la construction réelle

## Livrable — toujours dans la KB du client

Écrire le livrable daté dans
`./rapido-kb/startup/forge/ideation/ideation-mvp-wireframes.md` (le dossier du client,
dans SON répertoire de travail — jamais dans le plugin). Si des chiffres
clés en sortent, proposer de mettre à jour
`./rapido-kb/startup/business-plan/hypotheses.md` (source + date).
