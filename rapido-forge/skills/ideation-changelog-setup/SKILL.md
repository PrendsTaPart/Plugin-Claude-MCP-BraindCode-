# Changelog Setup

**Catégorie** : Idéation  
**Durée** : 30-45 min

## Pourquoi

Un changelog public montre que tu améliores constamment le produit. Ça construit la confiance et l'engagement.

## Objectif

Créer et maintenir un changelog public.

## Livrable attendu

Page changelog avec premiers updates

## Étape 0 — Contexte (obligatoire)

Charger `./rapido-kb/` — surtout `./rapido-kb/startup/` (vision, persona,
offre, hypothèses construits par `dossier-startup-360` et
`interview-business-plan`). La KB PRIME : ne jamais redemander une
information déjà validée, la reformuler pour confirmation. Les chiffres
réels viennent des MCP — jamais de mémoire ; tout chiffre sans source datée
porte la mention « hypothèse fondateur, confiance faible ».

## Étapes

1. **Choisir le format** — Page dédiée, widget, email
2. **Définir la structure** — Date, type, description
3. **Documenter les premiers updates** — Features, fixes, improvements
4. **Publier et promouvoir** — In-app notification
5. **Créer le process** — Qui écrit ? Quand publier ?

## Pro tips

- Écris pour les utilisateurs, pas pour les devs
- Inclus des captures d'écran/GIFs
- Célèbre les demandes utilisateurs implémentées

## Erreurs fréquentes

- Trop technique, pas user-friendly
- Pas de régularité
- Oublier les petites améliorations

## Données & serveurs MCP

- **RapidoCRM** (`rapidocrm`) — données réelles, lecture d'abord

## Livrable — toujours dans la KB du client

Écrire le livrable daté dans
`./rapido-kb/startup/forge/ideation/ideation-changelog-setup.md` (le dossier du client,
dans SON répertoire de travail — jamais dans le plugin). Si des chiffres
clés en sortent, proposer de mettre à jour
`./rapido-kb/startup/business-plan/hypotheses.md` (source + date).
