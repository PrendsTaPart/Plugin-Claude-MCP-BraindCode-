# Feedback Analysis

**Catégorie** : Idéation  
**Durée** : 45-60 min

## Pourquoi

Les premiers feedbacks sont une mine d'or pour améliorer ton produit. Chaque critique est une opportunité d'amélioration.

## Objectif

Analyser et synthétiser les premiers retours utilisateurs.

## Livrable attendu

Rapport feedback + plan d'action

## Étape 0 — Contexte (obligatoire)

Charger `./rapido-kb/` — surtout `./rapido-kb/startup/` (vision, persona,
offre, hypothèses construits par `dossier-startup-360` et
`interview-business-plan`). La KB PRIME : ne jamais redemander une
information déjà validée, la reformuler pour confirmation. Les chiffres
réels viennent des MCP — jamais de mémoire ; tout chiffre sans source datée
porte la mention « hypothèse fondateur, confiance faible ».

## Étapes

1. **Collecter les feedbacks** — Reviews, support, NPS, interviews
2. **Catégoriser par thème** — UX, features, prix, support
3. **Quantifier les patterns** — Fréquence de chaque feedback
4. **Prioriser les actions** — Impact × Effort
5. **Communiquer les changes** — Changelog public

## Pro tips

- Les plaintes sont des cadeaux
- Recontacte les utilisateurs mécontents après amélioration
- Fais du feedback loop un process régulier

## Erreurs fréquentes

- Ignorer les critiques négatives
- Réagir à chaque feedback individuel
- Pas de suivi après les changements

## Données & serveurs MCP

- **RapidoCRM** (`rapidocrm`) — données réelles, lecture d'abord

## Livrable — toujours dans la KB du client

Écrire le livrable daté dans
`./rapido-kb/startup/forge/ideation/ideation-feedback-analysis.md` (le dossier du client,
dans SON répertoire de travail — jamais dans le plugin). Si des chiffres
clés en sortent, proposer de mettre à jour
`./rapido-kb/startup/business-plan/hypotheses.md` (source + date).
